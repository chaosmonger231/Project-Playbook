Suricata Monitoring Tool

Suricata is commonly used to improve visibility into network traffic, protocol activity, and suspicious communication patterns. It can help organizations detect unusual DNS lookups, suspicious HTTP requests, risky TLS fingerprints, SMB file activity, and other network-level signs that may point to ransomware or broader malicious activity.

Network visibility Traffic inspection Protocol-aware detection Ransomware indicators
This page is a lightweight Suricata overview, not a full enterprise deployment guide. It is meant to help explain where Suricata fits and provide starter examples that can later be adapted to your own environment.

What Suricata is good for

Suricata is most useful when you want visibility into what is happening across the network. That includes:

Traffic-side visibility
  • Suspicious DNS lookups
  • Unusual HTTP requests and headers
  • TLS fingerprinting and connection patterns
  • Potential command-and-control communication
File and protocol visibility
  • SMB filename activity
  • Ransom note file naming patterns
  • Suspicious encrypted extensions over SMB
  • Heuristic file write behavior on the network

Why it matters in ransomware detection

Many ransomware events generate observable network behavior before or during impact. Suricata can help surface those signs earlier by watching traffic patterns, protocol activity, and known indicators instead of relying only on host-side evidence.

A simple way to think about Suricata: it helps you see what systems are doing across the network, which makes it valuable for spotting suspicious communications and file-related activity moving between devices.

Basic deployment notes

Suricata rules are commonly managed in rule files that use Snort/Suricata syntax. Depending on how your environment is set up, you may maintain standalone rule files or use update tooling to pull and apply custom sources before restarting the service.

Example deployment notes
  • Rules are typically stored in .rules files using Snort/Suricata syntax.
  • Some environments also manage custom sources through update tooling.
  • After adding or updating rules, Suricata generally needs to reload or restart.
Example commands (Linux)
sudo suricata-update add-source projectplaybook https://<our-name>/rules.tar.gz
sudo suricata-update
sudo systemctl restart suricata
Replace placeholders like <our-name> with your actual domain or host, and make sure your rules are being added to the correct Suricata source or file path for your environment.

Example Suricata detections

Below are example rule ideas that are often useful in ransomware-oriented monitoring. These are not meant to be treated as universal production-ready content without validation and tuning.

1. WannaCry killswitch DNS lookup

Detects DNS queries for the classic WannaCry killswitch domain.

suricata.rules
alert dns $HOME_NET any -> any 53 (
  msg:"Ransomware – Possible WannaCry killswitch DNS lookup";
  dns_query;
  content:"iuqerfsodp9ifjaposdfjhgosurijfaewrwergwea";
  sid:2000001; rev:1;
)
2. WannaCry killswitch HTTP Host header

Detects HTTP requests containing the killswitch domain in headers.

suricata.rules
alert http $HOME_NET any -> $EXTERNAL_NET any (
  msg:"Ransomware – WannaCry killswitch HTTP request";
  flow:established,to_server;
  content:"iuqerfsodp9ifjaposdfjhgosurijfaewrwergwea"; http_header;
  content:"Host"; http_header;
  sid:2000002; rev:1;
)
3. Android WannaLocker DNS lookup

Detects DNS queries for a domain associated with WannaLocker.

suricata.rules
alert dns $HOME_NET any -> any 53 (
  msg:"Ransomware – Android WannaLocker DNS lookup";
  dns_query;
  content:"biaozhunshijian.51240.com";
  sid:2000003; rev:1;
)
4. JA3 TLS fingerprint associated with ransomware

Detects a specific JA3 hash example associated with suspicious TLS clients.

suricata.rules
alert tls $HOME_NET any -> $EXTERNAL_NET any (
  msg:"Ransomware – TLS JA3 fingerprint from SSLBL";
  ja3_hash; content:"2d8794cb7b52b777bee2695e79c15760";
  sid:2000004; rev:1;
)
5. LockBit ransom note over SMB filenames

Flags SMB filenames matching common ransom note names.

suricata.rules
alert smb any any -> any any (
  msg:"Ransomware – ransom note over SMB";
  smb.filename; content:"Restore-My-Files.txt"; nocase;
  sid:2000005; rev:1;
)
6. Ransomware extension over SMB

Detects SMB filenames ending in suspicious encrypted extensions.

suricata.rules
alert smb any any -> any any (
  msg:"Ransomware – encrypted extension over SMB";
  smb.filename; pcre:"\\.(lockbit|funksec|encrypted)$";
  sid:2000006; rev:1;
)
7. LockBit onion ransom gate in HTTP Host header

Detects requests using a known onion host in HTTP headers.

suricata.rules
alert http $HOME_NET any -> $EXTERNAL_NET any (
  msg:"Ransomware – LockBit ransom gate onion host";
  flow:established,to_server;
  content:"lockbitkodidilol.onion"; http_header; nocase;
  sid:2000007; rev:1;
)
8. High-volume SMB encryption behavior

Basic SMB write indicator that can be useful as a candidate for rate or correlation logic.

suricata.rules
alert smb any any -> any any (
  msg:"Possible ransomware – SMB file write (candidate for rate correlation)";
  smb.command; content:"write"; nocase;
  sid:2000008; rev:1;
)

Practical notes

  • Keep custom rule identifiers organized to avoid collisions.
  • Validate rules in a safe environment before relying on them operationally.
  • Tune detections to your own environment so they are useful and not overly noisy.
  • Do not assume example rules are complete coverage by themselves.
Suricata is especially valuable when you want stronger network-side visibility during detection and monitoring, but it works best when paired with sensible alert tuning, logging, and a clear process for triage and escalation.
