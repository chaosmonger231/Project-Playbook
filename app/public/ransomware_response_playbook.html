Ransomware Response Playbook

This playbook gives small and mid-sized organizations a practical starting point for recognizing, preparing for, and responding to ransomware incidents. It focuses on ransomware-specific warning signs, supporting tools, communication planning, notification drafting, and immediate containment actions.

Ransomware context

Ransomware is malicious software that encrypts files or systems and then demands payment in exchange for a decryption key or supposed restoration of access. In many cases, the attack is not limited to encryption. Threat actors may also steal sensitive data and threaten to release it publicly if the ransom is not paid.

Small organizations, schools, and local agencies are often attractive targets because they may have limited staffing, slower response processes, and fewer dedicated security tools in place.

Common entry points
  • Phishing emails with malicious links or attachments
  • Compromised or misused remote access tools such as RDP or VPN
  • Unpatched internet-facing systems and vulnerable applications
Common warning signs
  • Sudden file renaming or unusual extensions
  • Ransom notes appearing on systems or shared drives
  • Unexpected backup deletion or log clearing
  • Large numbers of files being changed in a short period
Paying a ransom does not guarantee recovery, may support criminal activity, and may still leave the organization exposed to data theft, repeat attacks, or incomplete restoration.

How ransomware incidents commonly unfold

Ransomware incidents often progress in a pattern. Understanding the rough stages can help responders recognize when an alert may be part of a larger attack.

1. Initial access

The attacker gains access through phishing, exposed services, stolen credentials, vulnerable software, or abused remote access tools.

2. Execution and spread

Malicious code begins running, spreading, or establishing persistence. This phase may also involve credential theft, disabling defenses, deleting backups, and preparing for broader impact.

3. Encryption and extortion

Files are encrypted, ransom notes appear, and the organization may be pressured to pay. In some cases, stolen data is used as leverage in a double-extortion scheme.

Supporting tools and deployment overview

This playbook assumes the organization may use both host-side and network-side monitoring to catch ransomware-related activity earlier. Two example tools are Wazuh and Suricata.

Wazuh

Wazuh helps monitor host activity such as suspicious commands, backup deletion attempts, log clearing, ransom note creation, and suspicious file extensions.

Suricata

Suricata helps monitor network activity such as suspicious DNS requests, unusual HTTP behavior, risky TLS fingerprints, and file-related activity across the network.

Example deployment SOP

The following steps describe a simple process for deploying pre-configured ransomware-oriented rules for Wazuh and Suricata. These are starter examples and should be tested in a safe environment before production use.

Prerequisites

  • Wazuh Manager: installed and accessible, with agents already deployed to endpoints.
  • Suricata Sensor: installed and active, with access to the Suricata configuration and rules directories.
  • Administrative access: SSH or local access to the relevant servers.
Wazuh deployment
  • Upload the ransomware rules XML file to the Wazuh rules directory.
  • Validate the rules file syntax before relying on it.
  • Confirm the ruleset is referenced in the Wazuh configuration if required.
  • Restart Wazuh Manager so the rules are loaded.
Example Wazuh commands
scp local_ransomware_rules.xml ExampleUser@wazuh-manager:/var/ossec/etc/rules/

/var/ossec/bin/wazuh-logtest

sudo nano /var/ossec/etc/ossec.conf

sudo systemctl restart wazuh-manager
Suricata deployment
  • Upload the ransomware rules file to the Suricata rules directory.
  • Add the rules file to the rule-files section of the Suricata configuration if needed.
  • Validate the configuration before restart.
  • Restart Suricata so the new rules are applied.
Example Suricata commands
scp SuricataRansomware.rules ExampleUser@suricata-sensor:/etc/suricata/rules/

sudo nano /etc/suricata/suricata.yaml

sudo suricata -T -c /etc/suricata/suricata.yaml -v

sudo systemctl restart suricata
If configuration formatting errors occur, correct them before restart. Rule validation should always happen in a safe environment before production use.

Communication tree planning

Ransomware response is not only technical. A clear communication tree helps reduce confusion, speed up response, and make sure the right people are informed in the right order.

Questions to answer
  • Who receives the first alert or report?
  • Who is the backup if the primary person is unavailable?
  • Who can approve isolation or broader containment decisions?
  • What group activates after hours or during weekends?
Why it matters
  • Reduces delays during a live incident
  • Avoids duplicated or conflicting actions
  • Improves coordination between technical and leadership staff
  • Helps smaller teams respond more consistently
A simple top-down contact chart with names, roles, after-hours contact methods, and fallback contacts can be more useful than a complex document that no one checks during an emergency.

Containment and immediate next steps

During a suspected ransomware event, the goal is to reduce damage quickly while preserving enough information to continue investigation and recovery.

1. Remove the affected endpoint from the network

If an endpoint appears compromised, isolate it from the network as quickly as possible. This may be done through an agent-based action if supported, or manually by disconnecting the device.

2. Identify the suspicious process or activity

Use alert details, logs, and endpoint information to identify what is likely driving the activity. Review command lines, file changes, user activity, and related alerts on the same host.

3. Follow internal recovery procedures

Recovery steps may include checking backups, rebuilding systems, resetting credentials, validating scope, and reviewing whether other systems show similar signs of compromise.

If data theft, legal reporting requirements, or external partner obligations are involved, the organization should follow the appropriate legal, regulatory, and reporting processes in parallel with technical response.
