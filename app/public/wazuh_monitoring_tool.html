Wazuh Monitoring Tool

Wazuh is commonly used to improve visibility into endpoint, host, log, and file activity. It can help organizations detect suspicious commands, backup deletion attempts, log tampering, unusual file modifications, and other host-level signs that may point to ransomware or broader malicious activity.

Endpoint visibility Log monitoring File integrity monitoring Process activity
This page is a lightweight Wazuh overview, not a full enterprise deployment guide. It is meant to help explain where Wazuh fits and provide starter examples that can later be adapted to your own environment.

What Wazuh is good for

Wazuh is most useful when you want visibility into what is happening directly on systems and endpoints. That includes:

Host-side visibility
  • Suspicious commands and process creation
  • Attempts to delete backups or shadow copies
  • Event log clearing or tampering
  • Unexpected system changes
File-side visibility
  • Ransom note file creation
  • Suspicious encrypted file extensions
  • Mass file modification behavior
  • File integrity monitoring alerts

Why it matters in ransomware detection

Many ransomware events leave signs on the affected system before or during encryption. Wazuh can help surface those signs earlier by watching for destructive behavior on hosts rather than relying only on network indicators.

A simple way to think about Wazuh: it helps you see what a device or server is doing, which makes it valuable for spotting suspicious activity happening directly on the system.

Basic deployment notes

Wazuh rules and related detection logic are often managed through XML files. Depending on how your environment is set up, you may place rules on the manager, use shared configuration, or apply them locally to endpoints.

Example deployment notes
  • Custom rules and decoders are typically stored in XML files.
  • Active response actions may use Bash or PowerShell scripts.
  • Deployment location depends on whether the environment uses manager-side, shared, or local agent configuration.
Example commands (Linux)
sudo curl -L https://<your-domain>.com/playbook/local_rules.xml \
-o /var/ossec/etc/shared/<group>/local_rules.xml
sudo systemctl restart wazuh-manager
Make sure your Wazuh deployment model matches the path you are writing to. In practice, this may differ depending on whether you are updating the manager, shared agent config, or a local agent.

Example Wazuh detections

Below are example rule ideas that are often useful in ransomware-oriented monitoring. These are not meant to be treated as universal production-ready content without validation and tuning.

1. VSS shadow copies deletion

Detects shadow copy deletion commands, which are often used to make recovery harder.

local_rules.xml
<rule id="100600" level="12">
  <if_sid>61603</if_sid>
  <field name="win.eventdata.CommandLine" type="pcre2">(?i)vssadmin\.exe.*delete\s+shadows.*\/all.*\/quiet</field>
  <description>Possible ransomware: Volume Shadow Copies deleted</description>
  <mitre><id>T1490</id></mitre>
</rule>
2. Event log clearing

Detects suspicious log clearing activity, which may be used to hide malicious behavior.

local_rules.xml
<rule id="100602" level="10">
  <if_sid>61603</if_sid>
  <field name="win.eventdata.CommandLine" type="pcre2">
    (?i)wevtutil\.exe\s+cl\s+(security|system|application)
  </field>
  <description>Suspicious log clearing via wevtutil</description>
  <mitre><id>T1070</id></mitre>
</rule>
3. Backup deletion through WBADMIN

Detects attempts to remove or damage backups, which can be a major ransomware warning sign.

local_rules.xml
<rule id="100604" level="12">
  <if_sid>61603</if_sid>
  <field name="win.eventdata.CommandLine" type="pcre2">
    (?i)wbadmin\.exe.*(DELETE\s+SYSTEMSTATEBACKUP|delete\s+catalog)
  </field>
  <description>Possible ransomware: wbadmin backup deletion</description>
  <mitre><id>T1490</id></mitre>
</rule>
4. Ransom note file creation

Detects common ransom note filenames created in user directories.

local_rules.xml
<rule id="100605" level="12">
  <if_sid>554</if_sid>
  <field name="file" type="pcre2">
    (?i)C:\\Users\\.+\\(Desktop|Documents|Downloads).*(Restore-My-Files\.txt|README\.txt)
  </field>
  <description>Possible ransomware: ransom note file created in user profile</description>
  <mitre><id>T1486</id></mitre>
</rule>
5. Suspicious encrypted file extensions

Detects file creation or modification patterns that match suspicious encrypted extensions.

local_rules.xml
<rule id="100606" level="12">
  <if_sid>554</if_sid>
  <field name="file" type="pcre2">
    (?i)\.(lockbit|funksec|encrypted)$
  </field>
  <description>Possible ransomware: encrypted file extension detected</description>
  <mitre><id>T1486</id></mitre>
</rule>

Practical notes

  • Keep custom rule identifiers organized to avoid collisions.
  • Validate rules in a safe environment before relying on them operationally.
  • Tune detections to your own environment so they are useful and not overly noisy.
  • Do not assume example rules are complete coverage by themselves.
Wazuh is especially valuable when you want stronger host-side visibility during detection and monitoring, but it works best when paired with good logging, sensible alert tuning, and a clear process for triage and escalation.
